### 前置模块：全员通识——AI Native 研发认知拉齐
*   **面向对象**：全体试点团队成员 (L0-L3)
*   **培训目标**：建立统一的 AI Native 研发认知基础，消除不同层级间的知识鸿沟。
*   **核心内容**：
    1.  **大模型全景图**：大模型的发展历程、核心能力、行业现状及未来趋势展望。
    2.  **效能革命演示 (Live Demo)**：现场演示"一句话生成 PRD"、"Spec 驱动代码生成"、"AI 辅助测试"等典型场景。
    3.  **AI Native 研发范式概览**：介绍"人写 Spec，AI 写 Code"的核心理念，及各角色在新范式中的职责定位。
*   **微型实战**：无（本模块为认知铺垫）

### 模块一：Spec 驱动开发——需求到结构化规范
*   **核心技能领域**：需求分析、Spec 编写、技术可行性评估
*   **主要参与角色**：L1 产品/管理（主导）、L2/L3（评审指导）、L0（学习观摩）

*   **L1 核心技能**：
    *   **AI 辅助需求分析**：利用 LLM 进行头脑风暴，生成产品概念与痛点分析。
    *   **Spec 编写实战**：学习将用户故事转化为结构化 Spec 文档，掌握 Spec 模板使用。
    *   **ROI 评估**：评估 AI 工具在需求阶段的投入产出比。

*   **L2/L3 协同技能**：
    *   参与 Spec 评审，指导 L1 完善 Spec 的完整性、可测试性与技术可实现性。

*   **微型实战任务**：
    *   **任务描述**：为一个**独立的API接口**（如"用户信息查询接口"）或一个**数据处理功能**（如"Excel导入数据清洗"）编写完整的Spec文档。
    *   **边界限制**：功能范围明确，不超过3个核心业务流程，避免复杂度过高。
    *   **产出物**：至少**5个典型场景的Spec模板**（支撑4.2节《Spec驱动开发规范》交付），每个Spec包含：需求描述、输入输出定义、业务规则、异常处理、测试要点。

---

### 模块二：AI 辅助编码——从 Spec 到代码实现
*   **核心技能领域**：Spec 驱动开发、AI 辅助代码生成、代码评审
*   **主要参与角色**：L2 核心工程师（主导）、L0（执行辅助）、L3（评审把关）

*   **L2 核心技能**：
    *   **Spec 驱动编码实战**：严格遵循"人写 Spec，AI 写 Code"工作流，利用 AI（如 GitHub Copilot、Cursor）生成符合 Spec 的代码。
    *   **代码质量检查**：利用 AI 进行代码质量检查、安全漏洞扫描、性能优化建议。
    *   **单元测试生成**：利用 AI 自动生成高覆盖率的单元测试代码。

*   **L0 核心技能**：
    *   在 L2 指导下，利用 Copilot 生成样板代码，完成简单功能模块开发。
    *   学习代码解读与智能排错技巧。

*   **微型实战任务**：
    *   **任务描述**：基于模块一产出的Spec，利用AI工具生成对应的**功能代码**和**单元测试代码**。
    *   **边界限制**：聚焦单一功能实现，代码量控制在200-500行。
    *   **产出物**：至少**5个完整的代码示例**（支撑4.2节实验代码资产），每个示例包含：源代码、单元测试、README文档、代码评审报告。

---

### 模块三：AI 驱动的测试革新——质量智能化
*   **核心技能领域**：AI 驱动测试用例生成、自动化测试脚本编写、根因分析
*   **主要参与角色**：L2 测试工程师（主导）、L0（执行辅助）、L2 开发（协同）

*   **L2 测试核心技能**：
    *   **AI 驱动用例生成**：基于 Spec 和代码，利用 AI 批量生成思维导图格式的测试用例，覆盖正常流程、异常流程、边界条件。
    *   **自动化测试脚本生成**：利用 AI 辅助编写 Selenium/Playwright 自动化测试脚本。
    *   **根因分析 (RCA)**：利用 AI 对失败用例的日志进行分析，快速定位 Bug 根因。

*   **L0 参与方式**：
    *   在 L2 测试指导下，执行测试用例，学习使用 AI 辅助排查缺陷。

*   **微型实战任务**：
    *   **任务描述**：为模块二产出的代码，利用AI生成**完整的测试用例库**和**自动化测试脚本**。
    *   **边界限制**：测试范围限定在单一功能，测试用例数量10-20个。
    *   **产出物**：至少**3种场景的自动化测试脚本模板**（支撑4.2节实验代码资产），包含：测试用例库（思维导图格式）、自动化测试脚本、测试报告模板。

---

### 模块四：RAG 知识库构建——智能检索实战
*   **核心技能领域**：RAG 架构设计、向量化处理、混合检索优化
*   **主要参与角色**：L3 技术专家（主导）、L2（协同实现）、L0/L1（学习观摩）

*   **L3 核心技能**：
    *   **RAG 架构设计**：非结构化数据清洗、向量化策略、混合检索 (Hybrid Search) 调优。
    *   **检索效果评估**：使用 RAGAS 等工具评估检索质量。

*   **L2 协同技能**：
    *   实现 RAG 知识库的具体组件（如文档解析、向量存储、检索接口）。

*   **微型实战任务**：
    *   **任务描述**：构建一个**小型 RAG 知识库原型**（如：内部技术文档检索系统），包含数据清洗、向量化、检索三个核心环节。
    *   **边界限制**：数据量控制在100-500篇文档，使用供应商提供的脱敏数据或通用数据集。
    *   **产出物**：至少**2个完整的RAG实现**（支撑4.2节实验代码资产），每个实现包括：数据处理脚本、向量化代码、检索接口代码、效果评估报告。

---

### 模块五：Agent 编排架构——智能体协作
*   **核心技能领域**：Agent 设计、工具调用、多智能体协作
*   **主要参与角色**：L3 技术专家（主导）、L2（协同实现）、L0/L1（学习观摩）

*   **L3 核心技能**：
    *   **Agent 编排架构**：基于 LangChain/AutoGPT 的智能体开发，工具调用 (Tool Use) 与多智能体协作。
    *   **Agent 调试与优化**：Prompt 工程优化，工具链调优。

*   **L2 协同技能**：
    *   实现 Agent 所需的工具函数（如数据库查询工具、API 调用工具）。

*   **微型实战任务**：
    *   **任务描述**：构建一个**简单的Agent应用Demo**（如：智能客服Bot、代码审查助手），包含至少2个工具调用。
    *   **边界限制**：功能聚焦单一场景，交互流程不超过5轮。
    *   **产出物**：至少**3个可运行的Agent Demo**（支撑4.2节实验代码资产），每个Demo包括：Agent代码、工具函数、Prompt模板、运行效果演示。

### 模块六：ModelOps 与模型微调——底层技术深潜
*   **核心技能领域**：模型微调、RAGAS评估、私有化部署
*   **主要参与角色**：L3 技术专家（主导）、L2 运维（协同）
*   **适用条件**：**仅当试点团队有实际模型微调或私有化部署需求时开展，否则可作为选修模块**

*   **L3 核心技能**：
    *   **大模型原理理解**：Transformer 架构深度解析。
    *   **模型微调实战**：LoRA/QLoRA 微调实战，RAGAS 自动化评估体系搭建。
    *   **私有化部署**：模型量化、vLLM/TensorRT 推理加速。

*   **微型实战任务**：
    *   **任务描述**：在一个**简化的微调场景**（如：代码注释生成、特定领域问答）中，完成模型微调、评估、部署的全流程。
    *   **边界限制**：使用小规模数据集（100-1000条），单卡微调，避免资源消耗过大。
    *   **产出物**：**1-2个模型微调验证案例**（支撑4.2节实验代码资产），包括：微调脚本、评估报告、部署文档。

### 贯穿全程：标准资产汇总与SOP编写
*   **目标**：将各模块的微型实战产出汇总整合，形成完整的**AI Native 标准作业资产库**。
*   **活动方式**：在培训周期后期（建议最后1-2周），组织试点团队进行**资产整合工作坊**。

*   **产出物**（直接支撑4.2节验收）：
    1.  **《AI Native 工作流标准操作流程（SOP）》**：
        *   基于各模块实战经验，总结出各技能模块的**接入标准**和**协作接口**。
        *   包含流程图：如何从Spec到代码，如何从代码到测试，如何接入RAG/Agent能力。
        *   不少于20页，需通过采购方技术委员会评审。
    
    2.  **《企业级 Prompt 最佳实践》**：
        *   收集各模块实战中沉淀的 Prompt 工程示例（至少20个典型场景）。
        *   每个示例包含：场景描述、Prompt模板、预期输出、优化技巧。
        *   覆盖需求澄清、代码生成、测试用例生成、RAG检索、Agent调用等关键环节。
    
    3.  **《Spec 驱动开发规范》**：
        *   基于模块一的Spec模板库（至少5种典型需求类型）。
        *   质量检查清单（Check List），确保Spec的完整性、可测试性、技术可实现性。
        *   AI辅助Spec生成指南，降低Spec编写门槛。
    
    4.  **模块化代码资产库**：
        *   将模块二至六产出的所有代码示例、测试脚本、RAG实现、Agent Demo统一整理成**代码仓库**（Git格式）。
        *   包含完整的README、使用文档、部署指南。
        *   确保每个组件可独立运行，便于后续团队直接复用。

## 关键设计改进点

### 1. 解决时序矛盾
- ❌ 旧方案：阶段一学习需求 → 启动项目 → 等待阶段三才学开发 → **项目停滞**
- ✅ 新方案：模块一学Spec → 完成Spec微型实战 → 模块二学编码 → 完成编码微型实战 → **独立完成，无依赖**

### 2. 降低失败风险
- ❌ 旧方案：一个完整项目失败 → 全盘皆输
- ✅ 新方案：6个独立模块，即使某模块效果不佳，其他模块产出仍有价值

### 3. 提升资产质量
- ❌ 旧方案：一个可能无法完成的项目
- ✅ 新方案：10-20个经过实战验证的高质量组件（Spec模板5个+代码示例5个+测试脚本3个+RAG实现2个+Agent Demo3个+微调案例2个）

### 4. 与4.2节验收完美对齐
- ✅ 课件体系（30%）：各模块教学PPT+学员手册+Lab Guide
- ✅ 实验代码资产（25%）：模块化代码仓库（Spec案例≥5、代码示例≥5、测试脚本≥3、RAG实现≥2、Agent Demo≥3）
- ✅ 技术指导文件（30%）：SOP、Prompt实践、Spec规范
- ✅ 满意度（10%）：各模块学员反馈
- ✅ 工作流演示（5%）：资产整合工作坊演示
